# AutoVibe Gym — Local Dashboard

Локальная веб-панель для запуска LLM-агентов на AutoVibe Gym: выбрать модель +
тип прогона + датасет, настроить бюджет, запустить, наблюдать в реальном времени,
читать ноутбук/траекторию решения, видеть скор и ошибки, управлять датасетами и
моделями, сравнивать прогоны.

Это **отдельный** от `gym/` модуль. Бэкенд переиспользует тот же `.venv` проекта
(в нём уже есть FastAPI/uvicorn/MLflow/pandas) и запускает раннеры `experiments.*`,
читая их результаты из MLflow.

```
dashboard/
├── server/            FastAPI backend
│   ├── app/           роутеры (health, settings, datasets, models, runs) + сервисы
│   ├── requirements.txt
│   └── run.sh         запуск через .venv проекта
└── web/               Vite + React + TypeScript фронтенд
    └── src/           токены, UI-примитивы, экраны, API-клиент
```

## Запуск

**Бэкенд** (из корня репозитория):

```bash
dashboard/server/run.sh                 # → http://127.0.0.1:8000  (docs: /docs)
# или: .venv/bin/python -m uvicorn dashboard.server.app.main:app --reload --port 8000
```

**Фронтенд:**

```bash
cd dashboard/web
npm install
npm run dev                             # → http://localhost:5173  (проксирует /api → :8000)
```

## Где выполняется код (важно)

- **Локальный режим** (`run.sh` + `npm run dev` на ноутбуке): дашборд **запускает gym
  как локальный процесс**, и Jupyter-кернел с обучением моделей крутится на твоей
  машине. Удалённо вызывается **только LLM**. Поэтому ноутбук греется.
- **Серверный режим (рекомендуется)** — разверни весь дашборд на GPU-сервере: gym,
  кернелы и LLM работают там, а ноутбук лишь открывает UI в браузере. Нулевая
  локальная нагрузка. См. ниже.

## Вариант Б (рекомендуется, если на сервере закрыты порты): сайт локально, gym по SSH

Сайт работает на твоём маке, а каждый прогон выполняется **на сервере по SSH** — компьютер
не нагружается, и серверу не нужен открытый HTTP-порт (нужен только SSH). Локальный дашборд
сам запускает раннер на сервере, тянет артефакты обратно через `rsync` и рисует их.

Настройка в UI: **Настройки → «Выполнение на сервере (SSH)»**: включить тумблер и заполнить
`SSH (user@host)`, путь к репозиторию, python венва сервера, каталог прогонов. Кнопка
«Проверить связь» прозвонит сервер. Аутентификация — лучше по ключу
(`ssh-copy-id <user@host>` один раз), пароль — запасной вариант (хранится только локально).

Требования на сервере: актуальный код (эта ветка), подготовленные датасеты
(`scripts/prepare_datasets.py`) и тот же венв gym, что и обычно. Итоговый скор берётся из
сводки, которую раннер печатает в конце, — отдельный доступ к MLflow сервера не нужен.

## Запуск на GPU-сервере целиком (если порт на сервере доступен)

FastAPI умеет отдавать собранный фронтенд, поэтому весь дашборд — это один процесс.

```bash
# 1. На сервере: получить актуальный код (эта ветка) и подготовить датасеты
git fetch && git checkout dev/claude/web-dashboard && git pull
.venv/bin/python scripts/prepare_datasets.py          # если ещё не готовы

# 2. Поставить зависимости дашборда в серверный venv
.venv/bin/pip install -r dashboard/server/requirements.txt

# 3. Собрать фронтенд (нужен node). Если node на сервере нет — собери dist локально
#    и скопируй dashboard/web/dist на сервер, либо задай AUTOVIBE_DASHBOARD_WEB.
BUILD=1 PORT=8011 HOST=0.0.0.0 dashboard/server/serve.sh
```

С Мака открой `http://<LAN-IP сервера>:<PORT>` (напр. `http://10.8.45.1:8011`).
Все прогоны теперь идут на сервере; локально — только рендеринг UI.
`/api` фронтенд зовёт по тому же origin, прокси не нужен.

> Серверный venv должен содержать актуальный gym (эта ветка). Если на сервере была
> старая версия — шаг 1 её обновляет.

## Архитектура

- Прогоны = процессы `experiments.run_gym / run_baseline / run_multishot`, логируются
  в MLflow (`mlflow.db` + `mlruns/` в корне репо). Бэкенд читает их оттуда и из
  артефактов эпизода (`solution.ipynb`, `validation_trajectory.json`, `episode_summary.json`).
- 4 режима UI → 3 раннера: single→baseline, repeated→multishot,
  iterative→`run_gym --episode-mode iterative_no_checklist`, gym→`run_gym --episode-mode gym_with_checklist`.
- Датасеты управляются через общий ingestion/config слой:
  `datasets/<name>/{config.yaml,raw_data/,prepared/}`.
  Dashboard умеет:
  создавать конфиг, загружать raw-файлы, скачивать их по URL, настраивать
  raw vs pre-split режим, multi-table joins, preview/validate/prepare и потом
  запускать эксперименты на стандартном `prepared/` layout.
- Реестр моделей хранится в `dashboard/server/data/models.json` (у gym своего реестра нет).
